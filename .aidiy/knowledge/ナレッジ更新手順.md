# ナレッジ更新手順

## 目的

`.aidiy/knowledge` は、AiDiy2026 リポジトリ内で再利用できる修正知見を蓄積するためのプロジェクト専用フォルダである。
作業ログではなく、次回以降の実装・検証・レビューで参照できる手順、注意点、判断基準を残す。

## 更新前の確認

1. 担当ファイルを固定する。
   - 並列作業時は、親エージェントがサブエージェントごとに担当ファイルを明示する。
   - サブエージェントは担当ファイル以外を変更しない。
   - `_index.md` と `_最終変更.md` は親エージェントが最後に統合するため、サブエージェントは原則として触らない。
   - 作業前に `git status --short .aidiy/knowledge` で並列編集の有無を把握する。担当外ファイルの差分は触らない。
2. 既存ファイルがある場合は、現在の内容を読んで追記・統合する。
   - 同じテーマの重複ファイルを増やすより、既存ファイルへ整理して追記する。
   - 既存編集を戻さず、内容の衝突がある場合は追記側で整合する。
3. 文字コードは UTF-8 とし、日本語の見出しと本文で書く。
   - Markdown として読める構造にする。
   - 実装ファイル名、コマンド、パスはバッククォートで囲む。
4. ファイル操作を伴う本体修正の前は `_index.md` と `_最終変更.md` を参照する。
   - ただし、並列サブエージェントとしてナレッジだけを担当する場合は、参照に留め、指示がない限りこの2ファイルを更新しない。

## 書き方

- 先頭に目的や適用範囲を書く。
- 手順は、次回の作業者がそのまま実行できる順番にする。
- 「何をしたか」より「次にどう判断するか」を残す。
- 関連ファイルを書く場合は、実在するパスを使う。
- 古い状態の断定を避け、必要に応じて「現行実装を確認してから適用する」と書く。
- 大きな仕様変更と、`.aidiy/knowledge` の知見整理を混同しない。
- TODOだけの見出し、作業者名、時刻だけのログ、未確認の推測は残さない。
- 仕様値を書くときは同期元を併記する。例: JWT 60分は `backend_server/auth.py`、MCP 8サーバーは `backend_mcp/mcp_main.py`。
- 手順が環境依存の場合は、Windows PowerShell 前提か、bash 前提かをコードブロックの言語や本文で分ける。

## 並列作業時の分担

親エージェントは、作業開始前に次の単位で分担を固定する。

- 1 サブエージェントにつき 1 つ以上の明確な担当ファイルを割り当てる。
- 担当外ファイルの編集を禁止する。
- `_index.md` は、すべてのサブエージェント成果物を確認したあとに親エージェントがリンク整理する。
- `_最終変更.md` は、最後に親エージェントが全体の変更要約として更新する。
- サブエージェントは、担当ファイル内に完結する内容だけを書く。

この分担により、同時編集による競合を担当ファイル単位に閉じ込められる。

## 編集後の検証

編集後は、担当ファイルに対して短く確認する。

1. Markdown のリンク切れを確認する。
   - 相対リンクを書いた場合は、リンク先ファイルまたはディレクトリが存在するか確認する。
   - 存在確認できないリンクは、断定的なリンクではなくコード表記のパスに留める。
2. UTF-8 BOM を確認する。
   - BOM が混入していないことを確認する。
   - BOM が必要な理由がない限り、UTF-8 without BOM を維持する。
3. 古い表現を検索する。
   - 旧名称、廃止済みポート、存在しないファイルパス、過去の実装前提が残っていないか確認する。
   - 不明な場合は断定せず、「現行実装を確認する」と書く。
4. 担当外ファイルに差分が出ていないことを確認する。
   - `git diff --name-only -- .aidiy/knowledge` で変更対象を確認する。
   - サブエージェントは、許可されたファイル以外が出た場合は編集せず、最終報告で衝突可能性として伝える。

## 新規ナレッジ作成の判断

新規ファイルは、既存テーマに自然に統合できない場合だけ作成する。

- 既存テーマに1見出しで収まる内容は、既存ファイルへ追記する。
- 複数回参照される独立手順、チェックリスト、障害対処は新規ファイルにしてよい。
- ファイル名は日本語で、内容が分かる名詞句にする。
- `_index.md` へのリンク追加は、単独作業なら最後に実施する。並列作業のサブエージェントなら親エージェントへ追加候補として報告する。

## 自己検証の残し方

新規API・新規テーブル・新規画面を作った場合は、該当する docs のチェック観点を使って検証し、ナレッジには次回使える観点だけを残す。

- 「実行したコマンド一覧」だけではなく、どの失敗を防ぐための確認かを書く。
- バックエンドは `apps_models/__init__.py`、`apps_crud/__init__.py`、router登録、初期化処理の登録漏れをチェック観点として残す。
- フロントエンドは `npm run type-check`、対象画面の主要操作、Vite proxy 経由のAPI疎通をチェック観点として残す。
- MCP やブラウザ確認を使った場合は、使ったツール名と代替確認方法も書く。

## MCP バックアップ失敗時の扱い

MCP バックアップが使える場合は、編集前後の差分確認に利用する。
ただし、バックアップ取得や差分取得に失敗した場合でも、担当ファイルの通常確認を優先して作業を止めない。

- 失敗した MCP 名、対象ファイル、失敗概要を最終報告に短く残す。
- 代替として `git diff -- .aidiy/knowledge/<担当ファイル>` などで差分を確認する。
- バックアップ失敗を理由に、担当外ファイルを編集したり既存編集を戻したりしない。

## 親エージェントへの引き渡し

サブエージェントは、担当ファイルの変更内容と検証結果だけを報告する。
親エージェントは各担当ファイルを確認し、必要に応じて `_index.md` へリンクを追加し、`_最終変更.md` に全体要約を統合する。
