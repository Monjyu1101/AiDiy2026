# ナレッジ更新手順

> 文書: `共通,ナレッジ更新手順.md` | 実装: `.aidiy/knowledge/_index.md`

## このメモを使う場面

- `.aidiy/knowledge` の既存メモへ知見を追記する
- 新規ナレッジファイルを作るか、既存ファイルへ統合するか判断する
- 作業履歴や一時メモがナレッジに混ざらないよう整理する

## 基本方針

`.aidiy/knowledge` は、AiDiy2026 で再利用できる修正知見を残すためのプロジェクト専用フォルダである。日々の作業記録ではなく、次回以降の実装・検証・レビューでそのまま使える HowTo、判断基準、注意点だけを残す。

## 残す内容

- このメモを使う場面
- 関連ファイル
- 手順
- 判断基準
- 注意点
- 確認方法
- 現行仕様の同期元

## AGENTS 整理時の扱い

- `AGENTS.md`、下位 `AGENTS.md`、`_AIDIY.md`、`CLAUDE.md` の整理は [`共通,AGENTS整理手順.md`](./共通,AGENTS整理手順.md) を優先する。
- AGENTS 側は概要、構成、参照先に絞る。
- 起動、追加、検証、トラブル対応などの HowTo は `.aidiy/knowledge` へ転記してから AGENTS 側を短くする。
- `_AIDIY.md` と `CLAUDE.md` は同一内容の最小入口にする。

## 残さない内容

- 作業日時、作業者名、単発の変更履歴
- 「対応した」「対応済み」など一時的な表現
- 実行しただけのコマンド羅列
- 未確認の推測
- TODO だけの見出し
- 同じ意味の長い説明や背景説明
- アプリ本体の仕様変更と混同するメモ

必要な検証コマンドは、何を確認するためのものか分かる形で残す。

## 更新前の確認

1. 担当ファイルを固定する。
   - 並列作業時は担当ファイル以外を変更しない。
   - `_index.md` は親エージェントが最後に統合するため、サブエージェントは原則触らない。
   - `_最終変更.md` は履歴ではなく再修正時の入口として扱い、直近作業の要約を蓄積しない。
2. 既存ファイルを読む。
   - 同じテーマの重複ファイルを増やさず、既存ファイルへ統合する。
   - 既存編集を戻さず、内容の衝突がある場合は現行実装を確認して整合させる。
3. 文字コードは UTF-8 without BOM を維持する。
4. 本体修正を伴う場合は、作業前に `_index.md` と `_最終変更.md` を参照する。

## 書き方

- 日本語で簡潔に書く。
- ファイル名、コマンド、パス、設定キーはバッククォートで囲む。
- 手順は次回の作業者がそのまま実行できる順番にする。
- 「何をしたか」ではなく「次にどう判断するか」を書く。
- 仕様値を書くときは同期元を併記する。
  - 例: JWT 60分は `backend_server/auth.py`
  - 例: MCP 16サーバーは `backend_tools/tools_main.py`
- 古い状態の断定を避け、不明な場合は「現行実装を確認する」と書く。
- Windows PowerShell 前提と bash 前提は本文またはコードブロックで分ける。

## 並列作業時のルール

- 親エージェントがサブエージェントごとの担当ファイルを明示する。
- サブエージェントは担当ファイル内に完結する内容だけを書く。
- `_index.md` は全担当ファイルを確認したあとに親エージェントがリンク整理する。
- `_最終変更.md` は履歴ではなく再修正時の入口に限定する。直近作業の要約は残さない。
- 担当外ファイルに差分や問題を見つけた場合は編集せず、最終報告に残す。

## 新規ナレッジ作成の判断

- 既存テーマに 1 見出しで収まる内容は既存ファイルへ追記する。
- 複数回参照される独立手順、チェックリスト、障害対処は新規ファイルにしてよい。
- ファイル名は日本語で、内容が分かる名詞句にする。
- 並列作業のサブエージェントは、`_index.md` へのリンク追加候補を最終報告に書く。

## 編集後の検証

```powershell
$files = @(
  ".aidiy/knowledge/対象ファイル.md"
)

# BOM 確認
foreach ($file in $files) {
  $bytes = [System.IO.File]::ReadAllBytes((Resolve-Path $file))
  if ($bytes.Length -ge 3 -and $bytes[0] -eq 0xEF -and $bytes[1] -eq 0xBB -and $bytes[2] -eq 0xBF) { $file }
}

# 未整理表現は、仮置き見出し・一時対応文・単発履歴が残っていないか本文を確認する
```

相対 Markdown リンクを書いた場合は、リンク先が存在するか確認する。存在確認できないものはリンクではなくコード表記のパスに留める。

## MCP バックアップ失敗時

MCP バックアップが使える場合は、編集前後の差分確認に利用する。失敗しても作業を止めず、担当ファイルの通常確認を優先する。

- 失敗した MCP 名、対象ファイル、失敗概要を最終報告に短く残す。
- 代替として `git diff -- .aidiy/knowledge/<担当ファイル>` などで差分を確認する。
- バックアップ失敗を理由に、担当外ファイルを編集したり既存編集を戻したりしない。
