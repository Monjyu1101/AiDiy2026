# バックエンドの起動と再起動

## このメモを使う場面
- バックエンドのコード変更を反映させたい
- サーバーを再起動したい
- ポートが残留していてサーバーが起動できない

## 関連ファイル
- `_start.py` — 対話形式の起動スクリプト
- `backend_server/temp/reboot_core.txt` — core サーバー再起動トリガー
- `backend_server/temp/reboot_apps.txt` — apps サーバー再起動トリガー
- `backend_mcp/temp/reboot_mcp.txt` — mcp サーバー再起動トリガー

## 起動手順

### 起動方法

```bash
# 対話形式（推奨）
python _start.py

# 個別起動（--reload 付き、コード変更が即時反映）
cd backend_server && .venv/Scripts/python.exe -m uvicorn core_main:app --reload --host 0.0.0.0 --port 8091
cd backend_server && .venv/Scripts/python.exe -m uvicorn apps_main:app --reload --host 0.0.0.0 --port 8092
cd backend_mcp    && .venv/Scripts/python.exe -m uvicorn mcp_main:app  --reload --host 0.0.0.0 --port 8095
```

### `_start.py` の対話プロンプト

`python _start.py` を実行すると、以下の 4 つを順番に対話式で確認する。

| プロンプト | デフォルト | 内容 |
|----------|-----------|------|
| `バックエンド(mcp)       起動しますか?` | **Yes** | port 8095（`mcp_main:app`） |
| `バックエンド(core,apps) 起動しますか?` | **Yes** | port 8091（`core_main:app`）+ 8092（`apps_main:app`）|
| `フロントエンド(Web)     起動しますか?` | **Yes** | port 8090（`npm run dev`） |
| `フロントエンド(Avatar)  起動しますか?` | **No**  | port 8099（`npm run dev`）|

Enter キーでデフォルト選択。各サービスは `stdout/stderr` をマージして `[サービス名]` プレフィックス付きで同一ターミナルに混在表示される。起動後は各プロセスの出力が `QUIET_WAIT_SECONDS=20秒` 間静止するまで待機してからブラウザを開く（最大 `QUIET_MAX_WAIT_SECONDS=60秒`）。

### `_start.py` 起動後の再起動（--reload なし）

```bash
# Windows
echo. > backend_server/temp/reboot_core.txt
echo. > backend_server/temp/reboot_apps.txt
echo. > backend_mcp/temp/reboot_mcp.txt
```

各サーバーの内部ウォッチャースレッドが `temp/reboot_*.txt` を 1 秒ポーリングで監視し、ファイルを検知したら `os._exit(0)` で終了する。`_start.py` が子プロセスの終了を検知して同じコマンドで再起動する。

### ポート残留の解消

```bash
# 使用中の PID を確認して強制終了
netstat -ano | findstr :8091
taskkill /PID <pid> /F
# 8092・8095 も同様
```

## 起動方法の使い分け

| 方法 | `--reload` | コード変更反映 | 用途 |
|------|-----------|-------------|------|
| `_start.py` | なし | reboot ファイル作成が必要 | 通常運用 |
| 個別起動 | あり | 自動 | 開発中（頻繁に変更するとき） |

## 起動時に実行される処理

- `core_main.py` は C系/A系、`apps_main.py` は M/T/V/S系を担当する。
- `apps_main.py` は `database.Base.metadata.create_all(... tables=[...])` で apps 系テーブルだけを作る。新規 Model を追加したら、この tables 配列にも追加する。
- `startup_event()` の実行順序:
  1. `apps_crud.apply_schema_migrations(db)` — カラム追加・テーブル再作成などの既存DB補正
  2. `apps_crud.init_db_data(db)` — 初期データ投入（1件でも存在すれば skip）、C採番の既存DB補正
- `_start.py` 起動では `--reload` なしのため、Python ファイルを直しただけでは反映されない。`temp/reboot_core.txt` / `temp/reboot_apps.txt` を作る。
- 起動時にベースラインバックアップがバックグラウンド実行される。起動直後のログにはバックアップ関連メッセージも混ざる。

## import エラーのデバッグ

サーバーが起動できないときは、まず構文確認、次に import 連鎖確認の順で絞り込む。

```powershell
# 1. 構文確認（import は実行しない）
cd backend_server
.venv\Scripts\python.exe -m py_compile apps_main.py apps_crud\init.py apps_router\M商品.py

# 2. import 確認（router/schema/crud まで実際に読み込む）
.venv\Scripts\python.exe -c "import apps_main; import core_main"
```

- `py_compile` は構文エラーだけ拾う。`ImportError` や `AttributeError` は import 確認で出る。
- `startup_event()` は import 時には実行されないため、初期データとスキーマ補正の確認は実起動が必要。
- `apps_models/__init__.py` または `apps_crud/__init__.py` の import 漏れは import 確認で `AttributeError: module has no attribute` として出やすい。

## 注意点

- `_start.py` で起動したサーバーは `--reload` なし — コード変更後は reboot ファイルを作成するか個別起動に切り替える
- スキーマ変更後は DB ファイルにも変更を適用する必要がある（適用漏れはサーバー起動エラーの原因）
- `_stop.py` は存在しない — 停止は `Ctrl+C` またはポート解放で行う
- SQLite Browser / DBeaver が DB を開いたままだと `database is locked` エラーが発生する — 閉じてから起動
- `py_compile` は構文確認だけで import 依存の不足までは拾えない。新規 Router 登録後は import 確認も行う。

## 確認方法

```
http://localhost:8091/docs  # core が起動していれば Swagger が表示される
http://localhost:8092/docs  # apps
http://localhost:8095/aidiy_sqlite/sse  # mcp（SSE ストリームが返れば起動中）
```

Backend 変更後の最小確認:

```powershell
cd backend_server
.venv\Scripts\python.exe -m py_compile apps_main.py apps_crud\init.py
.venv\Scripts\python.exe -c "import apps_main; import core_main"
```
