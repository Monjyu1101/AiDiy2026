# クリーンアップ手順

> 文書: `共通,クリーンアップ手順.md` | 実装: `_cleanup.py`, `backend_server/temp/`

## このメモを使う場面
- `_cleanup.py` の削除対象や確認プロンプトを変更する
- 一時ファイル、バックアップ、依存物の整理範囲を確認する
- cleanup 後に起動時再作成されるディレクトリを判断する

## 関連ファイル
- `_cleanup.py`
- `backend_server/temp/`
- `backend_mcp/temp/`
- `backend_hermes/temp/`
- `backup/`
- `docs/開発ガイド/**/code_samples/_cleanup.py`

## 対象

- ルート `backup/`
- `backend_server/temp/`
- `backend_mcp/temp/`
- `backend_hermes/temp/`
- 各 `.venv` / `venv`
- `node_modules`
- `dist`
- `__pycache__`

## 実装方針

- `temp` は中身だけではなく、フォルダごと `remove_directory()` で削除する。
- `core_main.py` / `apps_main.py` / `mcp_main.py` は起動時に必要な `temp` を再作成する。
- `backend_hermes/temp` は CLI ツールの一時出力先なので cleanup 対象に含める。
- ルート `backup/` は `_cleanup.py` の選択項目に含め、存在する場合は既定 `y` で削除確認する。
- `logs` は中身削除を維持する。ログフォルダ自体を削除する場合は、ログ設定側の再作成処理を先に確認する。
- `_cleanup.py` の仕様を変えたら、docs の code_samples に古い説明が残っていないか確認する。

## 検証方法

```powershell
python -m py_compile _cleanup.py
```

削除挙動を確認する場合は、実プロジェクトではなく一時ディレクトリ配下に `backend_server/temp`、`backend_mcp/temp`、`backend_hermes/temp`、`backup` を作り、`cleanup_backend()` / `cleanup_backend_mcp()` / `cleanup_backend_hermes()` / `cleanup_backup()` を直接呼ぶ。

## 注意点

- 本番や共有環境で cleanup を実行する前に、削除対象の実体パスを確認する。
- ルート `backup/` は復旧材料になるため、削除確認プロンプトなしで消さない。
- 並列作業中は `_cleanup.py` と docs サンプルの担当者を確認し、担当外ファイルを編集しない。
