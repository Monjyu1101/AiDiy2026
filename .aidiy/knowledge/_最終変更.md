# 再修正時の確認入口

このファイルは変更履歴を蓄積しません。
既存ルールとの互換のためファイル名は `_最終変更.md` のまま残し、再修正依頼や差し戻し時に最初に見る HowTo を決める入口として使います。

## 使う場面

- 以前触った箇所の再修正を依頼されたが、関連する知見ファイルがすぐ分からない。
- 複数レイヤーにまたがる不具合で、確認順序を決めたい。
- `.aidiy/knowledge` へ追記する前に、既存ファイルへ統合できるか判断したい。

## 確認順序

1. 依頼文から対象レイヤーを切り分ける。
   - DB / API / Backend 起動
   - 認証 / token / パスワード
   - AI コア / Code CLI / MCP
   - frontend_web
   - frontend_avatar / Electron
   - ドキュメント / ナレッジ整理
2. [`_index.md`](./_index.md) の「迷ったときの入口」で最初に見る HowTo を決める。
3. HowTo の関連ファイルを実際に読んで、現行実装と一致しているか確認する。
4. 複数レイヤーにまたがる場合は、backend の API 契約から frontend の利用箇所へ順に確認する。
5. 修正後に知見を残す場合は、再利用できる判断基準だけを該当 HowTo に統合する。

## 残さない内容

- 日付つきの変更履歴
- 担当者や実行経緯
- 編集対象の列挙だけの記録
- 検証結果だけの記録
- 一度限りの退避メモ

検証コマンドや失敗対処を残す場合は、同じ問題を防ぐための確認方法として該当 HowTo に統合します。
