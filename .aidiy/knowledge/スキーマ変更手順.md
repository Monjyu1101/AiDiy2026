# スキーマ変更手順（Alembic なし・SQLite）

## このメモを使う場面
- テーブルへのカラム追加・削除・型変更をしたい
- テーブルそのものを削除したい
- スキーマ変更をサーバー起動時に自動適用したい

## 関連ファイル
- `backend_server/apps_crud/init.py` — `apply_schema_migrations()` に追記
- `backend_server/core_crud/init.py` — C系テーブルの場合はこちら

## 実装手順

### カラム追加（ADD COLUMN）

```python
def apply_schema_migrations(db: Session):
    result = db.execute(text("PRAGMA table_info(テーブル名)")).fetchall()
    existing = [row[1] for row in result]
    if "新カラム名" not in existing:
        db.execute(text("ALTER TABLE テーブル名 ADD COLUMN 新カラム名 TEXT"))
        # 既存行に初期値を入れたい場合
        db.execute(text("UPDATE テーブル名 SET 新カラム名 = 'デフォルト値' WHERE 新カラム名 IS NULL"))
        db.commit()
```

- SQLite の `DROP COLUMN` や型変更は制約が多いため、互換性を優先してテーブル再作成方式にする
- カラムを追加した直後に UPDATE で初期値を埋めておくとアプリ側でNULLチェック不要になる

### カラム削除・型変更（テーブル再作成）

```python
# 1. 新テーブルを作成（既存モデル定義から）
db.execute(text('CREATE TABLE IF NOT EXISTS "テーブル名_new" AS SELECT ...'))
# 2. データ移行
db.execute(text('INSERT INTO "テーブル名_new" SELECT col1,col2,... FROM "テーブル名"'))
# 3. 旧テーブル削除
db.execute(text('DROP TABLE IF EXISTS "テーブル名"'))
# 4. リネーム
db.execute(text('ALTER TABLE "テーブル名_new" RENAME TO "テーブル名"'))
db.commit()
```

### テーブル削除

```python
db.execute(text('DROP TABLE IF EXISTS "旧テーブル名"'))
db.commit()
```

### inspector による存在確認

```python
from sqlalchemy import inspect
inspector = inspect(db.bind)
if inspector.has_table("テーブル名"):
    # テーブルが存在する場合の処理
```

## 注意点

- `apply_schema_migrations()` は `init_db_data()` の**前**に呼ぶこと（テーブル構造が揃っていないと初期データ投入が失敗する）
- カラム名に日本語を使う場合はダブルクォートで囲む: `"新カラム名"`
- `db.commit()` を忘れると変更が反映されずサーバー起動エラーになる
- スキーマ変更後は既存DB（`database.db`）に手動適用が必要。未適用のままサーバーを起動するとエラーになる
- 新規テーブルは `create_all()` で作れるが、既存テーブルの列追加・PK変更・列削除は `create_all()` では反映されない。既存DBが残る運用では `inspect(db.bind)` / `PRAGMA table_info()` で差分を見て、起動時補正か手動SQLを必ず用意する。
- 複合PKを追加する変更（例: `明細SEQ` 追加）は SQLite の `ALTER TABLE ADD PRIMARY KEY` では対応できない。旧テーブルをリネーム、新テーブルを Model から作成、必要なデータを INSERT、旧テーブルを DROP する。
- 開発用の空テーブルや再投入可能なサンプルデータなら `DROP TABLE` → `models.<Model>.__table__.create(bind=db.bind, checkfirst=True)` でよい。実データがある場合は、旧データの移行SQLを明示してから DROP する。
- `inspector` はスキーマ変更後に古い情報を持つことがある。`ALTER TABLE` / `DROP TABLE` / `CREATE TABLE` 後に続けて列確認する場合は `inspector = inspect(db.bind)` を作り直す。
- 互換救済コードは永続化しすぎない。過去スキーマ救済が不要になったら削除候補にするが、既存ユーザーDBを壊す可能性がある場合は残す。

## 既存DB差分の判断基準

| 変更内容 | 推奨対応 |
|---------|----------|
| NULL許容カラム追加 | `ALTER TABLE ADD COLUMN` |
| NOT NULLカラム追加 | `DEFAULT` 付きで追加し、必要なら既存行を `UPDATE` |
| 表示用/任意項目追加 | 起動時補正で追加してよい |
| PK構成変更 | テーブル再作成 |
| カラム名変更 | テーブル再作成、または新列追加＋データコピー＋旧列は互換残し |
| 不要テーブル削除 | `DROP TABLE IF EXISTS` |
