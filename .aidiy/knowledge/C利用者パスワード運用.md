# C利用者パスワード運用

## このメモを使う場面
- `C利用者` の登録・変更・認証ロジックを修正する
- 初期データのパスワード保存方式を確認する
- 既存DBとの互換を保ちながらパスワード保存方式を改善する

## 基本方針
- 通常の利用者登録はハッシュ保存を前提にする。
- パスワード変更時は、更新対象をハッシュ保存へ切り替える。
- 認証側は、既存DB互換のためプレーン一致とハッシュ照合の両方に対応する。
- 初期データ投入時の扱いは現行実装を確認し、既存運用との互換を壊さない。

## 関連ファイル
- `backend_server/core_crud/C利用者.py`
- `backend_server/core_router/C利用者.py`
- `backend_server/tests/test_c利用者_password.py`

## 実装手順

### 1. 登録

- 画面やAPI経由で新規作成する利用者はハッシュ保存する。
- `create_C利用者(..., hash_password=False)` のような例外引数がある場合は初期データ投入用として扱い、通常 Router からは使わない。
- 一覧・取得APIのレスポンスにパスワード項目を含めない。

### 2. 変更

- パスワードを書き換える場合は、保存前に必ずハッシュ化する。
- パスワードを変更しない更新では、既存パスワードを再ハッシュしない。
- 入力が空・未指定の場合の扱いは Router 側で明確にする。

### 3. 認証

- `verify_パスワード()` は既存プレーン保存利用者とハッシュ保存利用者の両方を照合できるようにする。
- bcrypt 形式（例: `$2...`）は `bcrypt.checkpw()` で照合する。
- 認証失敗時は、外部応答で「ID不存在」と「パスワード不一致」を詳細に分けない。

## 注意点

- 既存DBにプレーンテキストのパスワードが残っていても、即時一括移行を前提にしない。
- 認証ロジックをハッシュ専用にすると、既存データでログイン不能が発生する。
- bcrypt ハッシュは毎回 salt が変わる。同じパスワードでも保存文字列一致では確認しない。
- 初期利用者の ID、初期パスワード、保存方式を変える場合は README、Docker README、テストを合わせて確認する。
- ログやエラー応答にパスワード値を出さない。

## 確認方法

```powershell
cd backend_server
uv run python tests/test_c利用者_password.py
```

確認観点:

- 初期データ相当の作成方式が現行実装と一致する。
- 通常作成でハッシュ保存される。
- パスワード変更後はハッシュ保存になる。
- 認証がプレーン保存・ハッシュ保存の両方で通る。
