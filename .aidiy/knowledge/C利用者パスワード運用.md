# C利用者パスワード運用

## このメモを使う場面
- `C利用者` の登録・変更・認証ロジックを修正するとき
- 初期データのパスワード保存方式を確認したいとき
- 既存DBとの互換を壊さずにセキュリティ改善したいとき

## 結論
- 初期データ投入時の利用者は、既存運用との互換のためプレーンテキスト保存を許容する
- 通常の利用者登録はハッシュ保存を前提にする
- パスワード変更時は、更新対象をハッシュ保存へ切り替える
- 認証側は、プレーン一致とハッシュ照合の両方に対応して段階移行を成立させる

## 関連ファイル
- `backend_server/core_crud/C利用者.py`
  - `hash_パスワード()`
  - `verify_パスワード()`
  - `create_C利用者(..., hash_password=True)`
  - `authenticate_C利用者()`
- `backend_server/core_router/C利用者.py`
  - `/変更`

## 実装時の判断基準
- 初期化処理の自由度を残したい場合は、初期データだけプレーン保存を維持してよい
- 画面やAPI経由で新規作成する利用者は、平文保存へ戻さない
- 変更APIでパスワードを書き換える場合は、保存前に必ずハッシュ化する
- 認証処理を変更する場合は、既存プレーン保存利用者のログイン可否を必ず確認する
- `verify_パスワード()` は平文一致を先に許容し、bcrypt 形式（`$2...`）なら `bcrypt.checkpw()` で照合する。段階移行中はこの二段構えを維持する。
- `create_C利用者(..., hash_password=False)` は初期データ投入用の例外として扱う。通常 Router からの作成で `False` を渡さない。
- パスワードを変更しない更新では、既存パスワードを再ハッシュしない。入力が空・未指定の場合の扱いを Router 側で明確にする。

## 注意点
- 既存DBにプレーンテキストのパスワードが残っていても、即時一括移行は前提にしない
- 一覧・取得APIのレスポンスへパスワード項目を戻さない
- 認証ロジックをハッシュ専用にすると、既存データでログイン不能が発生する
- 段階移行中は「登録」「変更」「認証」で保存方式が揃っているかを個別に確認する
- bcrypt ハッシュは毎回 salt が変わるため、同じパスワードでも保存文字列は一致しない。テストでは文字列一致ではなく `verify_パスワード()` で確認する。
- 初期利用者の `admin` は既存運用との互換が強い。初期パスワードや保存方式を変える場合は README / Docker README / テストも合わせて確認する。
- ログやエラー応答にパスワード値を出さない。認証失敗時も「ID不存在」と「パスワード不一致」を詳細に分けて外部へ返さない。

## 最低限の確認方法
- `uv run python tests/test_c利用者_password.py`
- 確認観点
  - 初期データ相当の作成でプレーン保存できる
  - 通常作成でハッシュ保存される
  - パスワード変更後はハッシュ保存になる
  - 認証がプレーン保存・ハッシュ保存の両方で通る

## 追記するときの観点
- 新しい保存方式を追加した場合は、既存プレーン保存利用者との互換条件を明記する
- 認証失敗が発生した場合は、保存方式・照合方式・対象APIを分けて原因を残す
- テストを追加した場合は、確認コマンドと対象ケースをこのメモにも反映する
