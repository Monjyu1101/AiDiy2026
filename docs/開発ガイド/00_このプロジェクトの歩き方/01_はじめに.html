このプロジェクトの歩き方

01. はじめに

このプロジェクトのゴール

AiDiyは「管理画面中心の業務アプリ」を素早く構築するためのフレームワークです。
環境構築 → 画面確認 → API確認 → テーブル追加 → 画面追加 の流れで進めます。
日本語のテーブル名・APIパス・JSONキーで、現場の言葉にそのまま合わせて開発します。

学習レベル目安

自分の現在地を確認して、次の目標を決めましょう。

レベル 内容
Lv.1 動かす 環境構築して起動できる
Lv.2 読む 構成図とフォルダ構成を理解できる
Lv.3 追う APIリクエストの流れを追える
Lv.4 写す 既存テーブルを参考に新テーブルを追加できる
Lv.5 作る 画面を追加してフル機能を実装できる
Lv.6 設計する 新機能の設計から実装まで一人でできる

3つの基本原則

このプロジェクトを貫く3つの原則です。

日本語ファースト

テーブル名、カラム名、APIパス、JSONキーは全て日本語。
フレームワーク固有の語句(router, request等)のみ英語。

シンプル構成

余計な抽象化をしない。コードを読めば処理が分かる。
Router → CRUD → Model の直線的な流れ。

AI活用前提

AIエージェントが読みやすい構造設計。
CLAUDE.mdとAGENTS.mdでプロジェクトの全容を把握可能。

プロジェクト概要

コンセプト

管理画面中心の業務アプリフレームワーク
配車管理、商品管理、在庫管理などの業務機能をテンプレートとして提供します。

技術スタック

使用している主要技術の一覧です。

技術 用途
FastAPI APIサーバー(バックエンド)
SQLite データベース(ファイルベース)
SQLAlchemy PythonのORM(DB操作)
Vue 3 + Vite フロントエンド(画面)
Pinia Vue状態管理(ストア)
ポイント

デュアルサーバー構成: Core(8091)とApps(8092)の2サーバーで分離運用

提供される機能

AiDiyは開発フレームワーク/テンプレートシステムとして以下のサンプル実装を提供しています。

分類 テーブル例 説明
C系 (Core/Common) C権限, C利用者, C採番 権限管理、ユーザー管理、ID採番
M系 (Master) M配車区分, M車両, M商品 マスタデータ管理
T系 (Transaction) T配車, T商品入庫/出庫/棚卸 トランザクション管理（配車、在庫）
V系 (View) V利用者, V車両, V商品推移表 複数テーブルのJOIN表示（生SQL使用、DB VIEWは不使用）
S系 (Scheduler) S配車_週表示, S配車_日表示 スケジュール表示
A系 (AI/Advanced) AIコア, A会話履歴 マルチベンダーAI対話（Claude, OpenAI, Gemini）
X系 (Experimental) Xテトリス, Xインベーダー, Xリバーシ フロントエンドのみの実験的機能

主要な設計パターン

このプロジェクトで採用している設計パターンの一覧です。

パターン 説明
デュアルサーバー Core API (8091) と Apps API (8092) の2サーバー構成
POST中心のAPI 全CRUDエンドポイントをPOSTメソッドに統一
DB VIEWs不使用 V系は生SQLクエリ（SELECT + LEFT JOIN）で実装
カスタムID生成 (C採番) AUTOINCREMENTを使わず、C採番テーブルで管理
監査フィールド標準化 全テーブルに登録/更新の8カラムを共通付与
No Alembic マイグレーションツール不使用。DB削除で再構築
qTublerシステム UIフレームワーク不使用のカスタムテーブルコンポーネント
Reboot機構 temp/reboot_*.txt でバックエンドサーバーを個別再起動
