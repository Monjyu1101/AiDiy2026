06_設計から実装への流れ

設計書を実装へ落とすときは、成果物をそのまま作業分解へ変換します。ここでは workspace/AiDiy2026 の実装順に沿って説明します。

作業分解（推奨）

ステップ主担当成果物
1. DB実装バックエンドモデル定義、初期化確認、SQL確認
2. API実装バックエンドschema / model / crud / router
3. API検証バックエンドSwagger UIで仕様確認
4. 画面実装フロントエンド一覧・編集画面、バリデーション、メッセージ
5. 結合確認双方画面操作とAPI結果の整合確認

バックエンド実装への接続

  • 次章: 03_明日のために！その３_バックエンド開発
  • API設計書を基準に request/response を固定する
  • 監査カラム設定を共通処理として統一する

フロントエンド実装への接続

  • 次章: 04_明日のために！その４_フロントエンド開発
  • 画面設計書の項目名・必須条件をフォームへ反映する
  • APIの status/message をそのまま通知に活用する

実装時に崩れやすいポイント

崩れやすい点対策
項目名の不一致設計書を1つの正としてコピー運用する
バリデーション差分フロント/バックで共通一覧を持つ
エラーハンドリングの齟齬error.code と message の対応表を固定する
監査項目の漏れ登録系/更新系をテンプレート化する

コミット単位の推奨

  1. 設計書更新のみ
  2. DB/モデル実装のみ
  3. API実装のみ
  4. 画面実装のみ
  5. 結合修正のみ

最終確認

  • [ ] 画面設計書の操作シナリオどおりに遷移する
  • [ ] API設計書どおりのJSONが返る
  • [ ] DB設計書どおりのカラムが保存される
  • [ ] 監査情報が登録/更新で正しく反映される
