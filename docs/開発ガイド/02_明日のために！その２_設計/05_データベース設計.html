05_データベース設計

DB設計は、画面項目とAPI項目の最終的な受け皿です。ここでは M商品 テーブル設計と運用前提を整理します。

テーブル方針

項目内容
テーブル名M商品
主キー商品ID
DB種別SQLite
文字コードUTF-8

業務カラムと監査カラム

  • 業務カラム: 商品ID、商品名、単位、商品備考
  • 監査カラム（登録）: 登録日時、登録利用者ID、登録利用者名、登録端末ID
  • 監査カラム（更新）: 更新日時、更新利用者ID、更新利用者名、更新端末ID

制約設計の要点

  1. 商品IDは主キー（重複不可、NULL不可）
  2. 商品名と単位は業務必須（NOT NULL）
  3. 監査カラムのID/日時/端末は必須
  4. 文字数上限はアプリケーション側でバリデーション

論理削除の設計原則

このプロジェクトでは物理削除（DELETE文）を使用しません。削除はすべて論理削除で行います。

項目内容
削除フラグ全テーブルに 有効 カラム（Boolean）を持つ
/削除 エンドポイントの実態有効 = False を設定するだけ（DELETEは発行しない）
デフォルト値新規登録時は 有効 = True で作成
一覧表示デフォルトは 有効 = True のレコードのみ返す
無効レコードの参照リクエストの 無効も表示 フラグを true にすると有効・無効の両方を返す

有効カラムの DDL 定義例

有効 BOOLEAN NOT NULL DEFAULT 1

削除 API のロジック（実装例）

# /削除 エンドポイントの処理（物理削除ではない）
item.有効 = False
# 更新監査フィールドも同時に更新する
db.commit()

注意点

  • M系（マスタ）の一覧 API は 有効 フィルタを持たない（全件返す）。論理削除対応は V系・T系の一覧で行う。
  • 削除済みレコードを画面に表示したい場合はリクエストに "無効も表示": true を付与する。
  • DB上からはレコードが消えないため、必要なら管理画面から復元（有効 = True に戻す）できる。

実装時の確認ポイント

確認項目観点
DDL設計書のカラム定義と一致しているか
有効カラム全テーブルに 有効 カラムが存在するか
APIレスポンス画面が必要な項目を不足なく返しているか
監査値登録/更新で想定通り自動設定されるか
検索性能一覧取得で必要に応じた索引設計を検討しているか
