03_内部設計_API設計

API設計では、画面操作をサーバー処理へ正確に対応付けます。入力仕様とレスポンス仕様を明確にすると、フロント/バック両方の実装が安定します。

M商品API一覧

エンドポイント用途備考
POST /apps/V商品/一覧一覧取得画面初期表示、再検索
POST /apps/M商品/取得詳細取得更新モード初期表示
POST /apps/M商品/登録新規登録商品ID重複チェックが必要
POST /apps/M商品/変更更新存在チェックが必要
POST /apps/M商品/削除論理削除確認ダイアログ後に呼び出し。物理削除ではなく 有効=False を設定する

共通仕様

  • 認証方式: JWT（Authorization: Bearer ...）
  • 正常系/業務エラーともに JSON 形式で返却
  • レスポンスは status, message, data/error で統一
  • 全 CRUD エンドポイントは POST メソッドのみ使用する

件数制限・論理削除フラグ（V系・T系一覧）

V系・T系の一覧エンドポイントは、リクエストボディとして ListRequest を受け取ります。M系・C系の一覧はリクエストボディなし（固定上限 10,000 件）です。

ListRequest スキーマ（apps_schema/common.py）

class ListRequest(BaseModel):
    件数制限: Optional[bool] = True   # True=最大1000件、False=無制限
    無効も表示: Optional[bool] = False  # False=有効レコードのみ、True=全件
    開始日付: Optional[str] = None
    終了日付: Optional[str] = None
    商品ID: Optional[str] = None
    商品分類ID: Optional[str] = None
    生産区分ID: Optional[str] = None

フィールドの意味

フィールドデフォルト意味
件数制限truetrue のとき最大 1,000 件（LIMITED_MAX_ITEMS）に制限する。false で制限なし。
無効も表示falsefalse のとき 有効=True のレコードのみ返す。true で論理削除済みレコードも含めて全件返す。

バックエンドの処理ユーティリティ（backend_server/list_controls.py）

LIMITED_MAX_ITEMS = 1000  # V系・T系の最大件数上限

# 件数制限の ON/OFF を取得
def is_limit_enabled(request) -> bool: ...

# 無効レコードを含めるかを取得
def should_show_inactive(request) -> bool: ...

# 有効フィルタを WHERE 句の conditions リストに追加
def append_active_condition(conditions, request, column_name): ...

# LIMIT 句の文字列と値を返す（件数制限 OFF なら None）
def get_limit_clause(request) -> tuple[str, Optional[int]]: ...

一覧APIのレスポンス例

{
  "status": "OK",
  "message": "V商品一覧を取得しました",
  "data": {
    "items": [ ... ],
    "total": 42,
    "limit": 1000   // 件数制限 OFF の場合は null
  }
}

M系一覧との違い

種別リクエストボディ上限件数有効フィルタ
M系・C系 一覧なし（引数なし）固定 10,000 件なし（全件返す）
V系・T系 一覧ListRequestデフォルト 1,000 件（制御可）デフォルトで 有効=True のみ

設計時に固定すべき項目

  1. 必須項目と文字数制約
  2. 重複チェック、存在チェックのタイミング
  3. 画面表示用の項目名（日本語キー）
  4. エラーコードと画面メッセージの対応
