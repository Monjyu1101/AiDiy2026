# AiDiy 自己紹介動画シナリオ

入力元: `frontend_web/public/X自己紹介/index.html`

想定尺: 約100秒  
形式: `document_to_html` の初回試作用。音声付き `index.html` へ変換する前提。

## 全体方針

- 既存の `X自己紹介` ページの内容を、動画向けに10シーンへ圧縮する。
- 画面上の情報量は絞り、ナレーションで補う。
- 画像は「実画面風」「構成図」「業務サンプル」「AI/Avatar」「MCP Hub」の5系統に分ける。
- MP4化はまだ考えず、まずは音声付きHTMLで自然に再生できる構成を目指す。

## シーン構成

### scene_001: オープニング

表示:

```text
AiDiy
日本語で設計し、音声でもコードでも動かす
```

ナレーション:

> AiDiy は、日本語を第一言語にしたフルスタック業務システム開発フレームワークです。

狙い:

- 名前とコンセプトを最初に固定する。
- 既存ページの Hero アニメーションを動画の導入として使う。

### scene_002: 日本語ファースト

表示:

```text
Database / API / Frontend / Code
業務語彙と実装名を揃える
```

ナレーション:

> 特徴の中心は、日本語ファースト設計です。テーブル名、API、JSON キー、画面名、コード上の変数名まで、業務語彙と実装を直接対応させます。

狙い:

- AiDiy の差別化軸を明確にする。
- 日本語識別子が単なる見た目ではなく、業務理解と保守性のためであることを伝える。

### scene_003: マルチサーバー構成

表示:

```text
backend_server / backend_mcp / frontend_web / frontend_avatar
```

ナレーション:

> 構成は、Core API、Apps API、MCP サーバー、Web UI、Avatar UI に分かれています。分かれていても、Vite proxy と WebSocket で、ひとつの体験としてつながります。

狙い:

- 技術構成を俯瞰させる。
- 複数サーバー構成が複雑さではなく、役割分担であることを伝える。

### scene_004: 実務サンプル

表示:

```text
配車管理 / 生産管理 / 資材在庫管理
```

ナレーション:

> テンプレートの核は、実際に動く業務サンプルです。配車、生産、資材在庫管理を通じて、マスタ、トランザクション、ビュー、スケジューラの実装パターンを確認できます。

狙い:

- AiDiy が単なるデモではなく、業務システムの雛形であることを伝える。
- M/T/V/S 系の構造を自然に紹介する。

### scene_005: AI + 音声 + アバター

表示:

```text
Chat / Voice / Image / Avatar
```

ナレーション:

> AiDiy は、チャットだけではありません。音声入力、AI 音声応答、画像、VRM アバター、VRMA モーションを組み合わせて、対話型の AI 体験を作ります。

狙い:

- Avatar と音声の存在を印象づける。
- `frontend_avatar` の価値を紹介する。

### scene_006: MCP Hub

表示:

```text
MCP Hub × 11
observe / generate / verify
```

ナレーション:

> ポート 8095 には、11 個の MCP サーバーが同居します。ブラウザ操作、画面キャプチャ、DB 観測、ログ確認、型チェック、バックアップ、画像生成、音声認識、音声合成を、AI エージェントから利用できます。

狙い:

- 既存 MCP 群の強さを伝える。
- 後続の `document_to_html` でも画像生成・音声生成・自動再生に既存 MCP を使う流れにつなげる。

### scene_007: Code CLI と Hermes

表示:

```text
Multi Code CLI
Claude / Copilot / Codex / Gemini / OpenCode / Hermes
```

ナレーション:

> コード支援は、複数の Code CLI を並走できます。Claude、Copilot、Codex、Gemini、OpenCode、そして AiDiy 独自の aidiy_hermes を、用途ごとに切り替えられます。

狙い:

- Code AI パネルの意味を伝える。
- `aidiy_hermes` が単なる外部依存ではなく、AiDiy 側の独自拡張であることを出す。

### scene_008: 自己改善機構

表示:

```text
edit → observe → verify → backup
.aidiy/knowledge
```

ナレーション:

> 修正して終わりではありません。AiDiy は、変更、観測、検証、バックアップの流れを記録し、.aidiy/knowledge に再利用できる知見を積み上げます。

狙い:

- 「使うほど育つ」というメッセージを中心にする。
- AI エージェントが次回以降の作業精度を上げる仕組みとして見せる。

### scene_009: 広がるテンプレート

表示:

```text
MCP拡張 / CLI追加 / 業務領域の横展開
```

ナレーション:

> 今後は、MCP の追加、Code CLI の追加、業務領域の横展開によって、会計、HR、CRM などにも同じ設計を広げられます。

狙い:

- Future Roadmap を短くまとめる。
- 既存の配車・生産・在庫から、より広い業務領域へ拡張できることを示す。

### scene_010: クロージング

表示:

```text
AiDiy
日本語で、業務とAIをつなぐ。
```

ナレーション:

> AiDiy は、日本語で業務と AI をつなぐための実験基盤です。まずは Web UI、AI コア、Avatar から触ってみてください。

狙い:

- 最後にプロダクト名と価値を再提示する。
- `AiDiy を開く` の導線に接続する。

## 画像生成プロンプト方針

- `scene_001`: タイトル画面。サイバー調だが情報は少なめ。
- `scene_002`: 日本語 DB/API/UI/Code の対応図。
- `scene_003`: サーバー構成図。
- `scene_004`: 配車、生産、在庫の業務画面風。
- `scene_005`: Avatar と音声波形、チャット UI。
- `scene_006`: MCP Hub のハブ図。
- `scene_007`: 複数 Code CLI の端末パネル。
- `scene_008`: 自己改善ループ図。
- `scene_009`: ロードマップ図。
- `scene_010`: クロージング画面。

## 次に確認すること

1. 100秒は長すぎないか。
2. ナレーション量が TTS で自然に収まるか。
3. シーンごとの画像を AI 生成にするか、既存 HTML のスクリーンショットにするか。
4. `document_to_html` の最初の HTML は、スライド型にするか、縦スクロール型にするか。
5. このシナリオ粒度でスキル化できるか。
